# Workstream 10: Synthetic Data Generation

Create scripts that generate realistic synthetic receipt and pantry data. This allows experimentation with analytics and the full application without needing weeks of real purchase data.

## Context

Real data takes time to accumulate. If you want to analyze purchase patterns over 3 months, you need 3 months of real receipts. Synthetic data solves this by generating realistic-looking data programmatically.

Synthetic data is used extensively in the real world for:
- **Testing**: Verify your app works with large datasets before real data exists
- **Privacy**: Share realistic data without exposing real purchase history
- **Simulation**: Model scenarios ("what if I shopped differently?")
- **Education**: Learn analytics techniques with rich datasets

The key to good synthetic data is making it **realistic** — not random. Real shopping follows patterns: weekly staples, seasonal items, store preferences, and budget constraints.

## Instructions

1. **Analyze existing real data** to establish patterns:
   - Read the existing receipt and pantry data from the database
   - Identify and print:
     - Most common food items purchased and their frequencies
     - Typical price ranges by category
     - Typical quantities purchased
     - Store names and how often each is visited
     - Common category distributions (what fraction of items are produce vs dairy vs protein, etc.)
     - Date ranges and shopping frequency (trips per week)
   - These real patterns become the blueprint for generation
   - **If no real data exists**, use sensible defaults based on typical American grocery shopping

2. **Create a synthetic data configuration** at `config/synthetic_data.json` (NOT in Python code):

   **All synthetic data parameters should live in a JSON config file, following the project's configuration-over-code principle.** The Python generation scripts read this config. Students customize what gets generated by editing JSON, not Python.

   The config file should define:
   - **Food items by category** with realistic attributes:
     ```json
     {
       "food_items": {
         "protein": [
           {"name": "chicken breast", "price_range": [5.99, 9.99], "unit": "pounds", "typical_qty": [1, 3], "frequency": "weekly"},
           {"name": "ground beef", "price_range": [4.99, 8.99], "unit": "pounds", "typical_qty": [1, 2], "frequency": "weekly"}
         ],
         "vegetable": [...],
         "dairy": [...]
       }
     }
     ```
   - **Store profiles**: each store has a name, typical item categories (e.g., Trader Joe's skews organic, Costco has bulk sizes), and price adjustment factor
   - **Shopping patterns**: typical items per trip range, trips per week, day-of-week weights
   - **Seasonal adjustments**: optional — items that are more common in certain months
   - Base the item list on actual data from step 1 where available, and supplement with `config/normalization_mappings.json` food lists
   - Also create `src/synthetic/config.py` with a function `load_synthetic_config()` that reads the JSON config file and provides typed access to the data
   - Educational comments explaining:
     - Why configuration is in JSON, not Python (students can customize without code)
     - Why configuration is separate from generation logic (modularity)
     - How probability distributions create realistic variation
     - The concept of a "random seed" for reproducible generation

3. **Create a receipt generator** at `src/synthetic/generate_receipts.py`:
   - A function `generate_receipts(num_weeks: int = 4, trips_per_week: int = 2, start_date: date = None, seed: int = 42) -> list[Path]`:
     - For each simulated week:
       - Determine how many shopping trips (with slight random variation around `trips_per_week`)
       - For each trip:
         - Pick a store (weighted by preference)
         - Pick a day of the week (weighted by shopping day preferences)
         - Generate a basket of items:
           - Always include "weekly staples" (milk, bread, eggs, etc.) — items with `frequency: "weekly"`
           - Randomly include occasional items based on their frequency weights
           - Adjust quantities with small random variation
           - Apply store-specific pricing (with small random variation for realism)
           - Basket size should be realistic (10-25 items per trip)
         - Create a CSV matching the format in `data/receipts/` (same columns as real receipts)
         - Name the file: `synthetic_{store}_{date}.csv`
     - Save all CSVs to `data/synthetic/receipts/`
     - Print a generation summary:
       ```
       Generated 8 shopping trips over 4 weeks:
         - Trader Joe's: 5 trips, 87 items, $342.15 total
         - Safeway: 3 trips, 52 items, $198.40 total
       Total: 139 items across 8 trips ($540.55)
       Files saved to data/synthetic/receipts/
       ```
   - Educational comments about:
     - Using `random.seed()` for reproducibility (same seed = same data every time)
     - Weighted random choices for realistic selection
     - Why small random variation makes data feel real (prices fluctuate, quantities vary)
   - Make runnable directly

4. **Create a pantry snapshot generator** at `src/synthetic/generate_pantry.py`:
   - A function `generate_pantry_snapshot(snapshot_date: date = None, fullness: float = 0.7, seed: int = 42) -> Path`:
     - `fullness` controls what fraction of possible items are in stock (0.0 = empty kitchen, 1.0 = fully stocked)
     - Logic:
       - Start with the full food item list from config
       - Randomly select `fullness * len(items)` items to be "in stock"
       - Weight selection toward weekly staples (they're more likely to be present)
       - For each selected item:
         - Generate a realistic quantity (partially used items have less than full quantity)
         - Assign a realistic location (fridge, freezer, pantry, counter based on item type)
         - Assign a condition (good, opened, wilting for old produce, frozen for freezer items)
         - Use informal/short names (as a real pantry scan would): "milk" not "Organic Whole Milk 1gal"
       - Create a CSV matching `data/pantry/` format
     - Save to `data/synthetic/pantry/`
     - Print: items generated, categories represented, storage locations
   - Educational comments about modeling state (snapshot) vs events (transactions)
   - Make runnable directly

5. **Create a master generation script** at `src/synthetic/generate_all.py`:
   - A function `generate_all(num_weeks: int = 4, trips_per_week: int = 2, pantry_fullness: float = 0.7, seed: int = 42, run_pipeline: bool = True)`:
     - Runs the receipt generator
     - Runs the pantry snapshot generator
     - Optionally runs the full ingestion pipeline (ingest → normalize → match)
     - Prints a summary of everything generated
   - Accept command-line arguments for configuration
   - Educational comments about orchestration scripts
   - Make runnable directly with:
     ```bash
     uv run python -m src.synthetic.generate_all --weeks 8 --trips 2 --fullness 0.7
     ```

6. **Create a synthetic vs real comparison** at `src/analytics/synthetic_vs_real.py`:
   - If both real and synthetic data exist in the database:
     - Compare item frequency distributions
     - Compare price distributions
     - Compare category breakdowns
     - Create side-by-side matplotlib charts
   - If only synthetic data exists, just analyze the synthetic data quality:
     - Are the distributions realistic?
     - Are there obvious patterns that wouldn't exist in real data?
   - Educational comments about validating synthetic data
   - Make runnable directly

7. **Run the synthetic data generator** to create test data:
   ```bash
   uv run python -m src.synthetic.generate_all --weeks 4 --trips 2 --fullness 0.7
   ```

8. **Run the full pipeline** on the combined data (real + synthetic):
   ```bash
   uv run python -m src.analytics.overview
   ```

9. **Create documentation** at `docs/guides/synthetic_data.md`:
   - Why synthetic data is useful
   - How the generation works (patterns, randomness, seeds)
   - How to configure: change items, stores, prices, patterns
   - How to run with different parameters
   - How synthetic data is stored (separate directory from real data)
   - How to distinguish synthetic from real data in the database

## Things to Try After This Step

- Generate 12 weeks of data and look at the purchase pattern trends in the analytics notebooks
- Try different seeds (`--seed 123` vs `--seed 456`) and compare the generated data
- Generate a very full pantry (`--fullness 0.95`) and see how many recipes become makeable
- Generate a nearly empty pantry (`--fullness 0.2`) and see the shopping list grow
- Add new food items to `src/synthetic/config.py` (your favorites that aren't in the default list)
- Generate data for a specific diet by modifying the config (remove meat items for vegetarian)
- Run the synthetic vs real comparison to see how well the generated data mimics your real shopping
- Ask Claude Code: "Create a time-series chart showing weekly grocery spending over the synthetic 12 weeks"
