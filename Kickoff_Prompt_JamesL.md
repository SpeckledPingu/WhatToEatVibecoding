The application that I want to build for a demo is a "food inventory" and "what can I make?" recipe app. This app will have 3 different data sources: 
 - json from a browser plugin extracting and containing the information for recipes
 - tabular data from taking pictures of reciepts to know what was recently purchased
 - tabular data from taking pictures of the pantry where an llm extracts food items and information about the food items into a table to quickly inventory existing ingredients

These will be stored in a set of sqlite tables to track inventory and recipes over time. A future extension of this could be for the students to track when inventory was taken or when purchases at a store were provided to keep an active inventory as well as track food usage over time (frequently purchased food items). The recipes will also come with a "warm/cold weather" field and "rainy/sunny/cloudy" field for categorizing the "feel" of the food. Then they can use a weather API to get recommendations of recipes that can be made with the current inventory.

Additionally, the food items will have a "max expiration date" so that recipes that use close-to-expired food can be used up before they go bad. This will be a field where there is the "inventoried" date and then a table of food items and the number of weeks the food will be good for will create the time delta to add to the "active inventory" table. 

I (or the students) will supply recipes, one per file, in json format, along with a markdown formatted recipe. This shows them two different ways of representing the same data. These will be ingested into the same table.

Similarly, I (or the students) will provide a csv of a pantry inventory that will have its own schema and a separate csv of reciept data. The schemas will not be a one to one match, so they will need to be in their own table. Normalization scripts for joins and transformations to create a unified inventory view will need to be developed when that data is supplied.
